# Cheiron Agent for clinical trails visualization:

Cheiron turns a natural-language question about clinical trials into a **structured
visualization specification**, computed from live [ClinicalTrials.gov](https://clinicaltrials.gov/data-api/api)
data. You ask "How have pembrolizumab trials changed per year since 2018?" and you
get back a `time_series` spec with real per-year counts, the filters that were
applied, and citations back to the trials behind each number.

There's also a small static frontend (served at `/`) that renders every chart type,
including a 3D force-directed network graph — but the backend and its structured
output are the real deliverable.

---

## The one design decision everything else follows from

**The LLM only writes the plan. It never produces a number, a count, an NCT ID, or
any fact about a trial.**

The model's entire job is to translate the question into a small, validated
`AnalysisPlan` object (intent, entities, group-by dimension, measure, filters). From
there it's all deterministic Python: the plan is compiled into a ClinicalTrials.gov
query, the API is called, the records are aggregated locally, and a chart spec is
assembled. Every value you see in the output was counted by code from an API
response, not generated by a language model.

I made this the hard boundary because it's the only way I'd trust the numbers. A
model that's allowed to say "there are 41 Phase 3 trials" will eventually make that
number up. A model that can only say "group by phase, count studies" can't — the
count comes from somewhere auditable. The deep-citations feature is the same idea
taken one step further: every datum can point you at the exact trial records and
field values it was built from.

## How it works

Two timelines:

- **Startup (once):** introspect the API — pull `/studies/enums`, `/studies/metadata`,
  `/studies/search-areas`, and `/version`, and cache them. This gives me the live enum
  whitelists (so I can normalize "Phase II" → `PHASE2`), the field→JSON-path map, the
  valid `AREA[...]` names, and the data timestamp. If any of those calls fail, it falls
  back to the hardcoded enums from the brief and logs which path it used.

- **Per request (six stages):**
  `interpret → validate → compile → fetch → aggregate → assemble`
  1. **interpret** — LLM (forced tool-use) or a no-LLM rule fallback → `AnalysisPlan`
  2. **validate** — enforce caps, normalize enum values, apply structured-field overrides
  3. **compile** — plan → `/studies` request params (a _list_, one per comparison arm)
  4. **fetch** — paginated, polite httpx client
  5. **aggregate** — count/sum per bucket, keep the contributing NCT IDs
  6. **assemble** — the selector picks the chart type and shapes the data; citations attach; `meta` is built

## Request / response schema

A frontend engineer shouldn't have to guess, so the schemas are Pydantic models and
also show up in the live OpenAPI docs at `/docs`.

### Request (`POST /visualize`)

```jsonc
{
  "query": "string, required", // the natural-language question
  "condition": "string?", // optional structured fields; when present
  "drug_name": "string?", //   they are TREATED AS GROUND TRUTH and
  "sponsor": "string?", //   override whatever the LLM extracted
  "country": "string?",
  "phase": "string | string[]?", // normalized against the enum whitelist
  "status": "string | string[]?",
  "intervention_type": "string | string[]?",
  "start_year": "int?",
  "end_year": "int?",
  "compare": "string[]?", // if present, forces a comparison
  "max_results": "int?",
  "include_citations": "bool = true",
}
```

### Response

```jsonc
{
  "visualization": {
    "type": "bar_chart | grouped_bar_chart | time_series | histogram | scatter_plot | network_graph | stat",
    "title": "string",
    "encoding": { "x": "...", "y": "...", "series": "...", "color": "..." },  // channels → fields
    "data":  [ /* chart rows, each optionally with citations[] + citation_count */ ],
    "nodes": [ /* network only */ ],
    "edges": [ /* network only */ ]
  },
  "meta": {
    "interpretation": { /* the plan the LLM produced */ },
    "filters_applied": { /* the actual query.*/filter.* params sent */ },
    "source": "ClinicalTrials.gov API v2",
    "data_timestamp": "...",
    "total_matched": 0, "truncated": false,
    "missing_count": 0, "skipped_count": 0,
    "measure": "study_count", "units": "studies",
    "sorting": {...}, "grouping": {...}, "time_granularity": "year",
    "assumptions": [ "..." ],
    "geo": true   // hint flags added by the selector when relevant
  }
}
```

## Visualization coverage

The **selector**, not the LLM, decides the final chart type from the plan — so the
mapping is deterministic and testable:

| Question shape                                    | Type                                                              |
| ------------------------------------------------- | ----------------------------------------------------------------- |
| distribution over a category                      | `bar_chart`                                                       |
| distribution split by a second category           | `grouped_bar_chart`                                               |
| A vs B over a shared dimension                    | `grouped_bar_chart` (or `time_series` if the shared axis is year) |
| trend over time                                   | `time_series`                                                     |
| by country                                        | `bar_chart` + a `geo` hint                                        |
| top-N ranking                                     | sorted `bar_chart`, top N only                                    |
| distribution of a continuous measure (enrollment) | `histogram`                                                       |
| relationship between two continuous measures      | `scatter_plot`                                                    |
| a single number                                   | `stat`                                                            |
| co-occurrence / relationships between entities    | `network_graph`                                                   |

The network graph is a real one, not a toy: it builds weighted **sponsor↔drug**,
**drug↔drug**, **sponsor↔condition**, or **drug↔condition** graphs (node weight =
studies containing it, edge weight = studies supporting the pair), de-duplicates
pairs per study, and caps to the top-weighted N nodes. Group-by dimensions live in a
small registry (`phase`, `start_year`, `overall_status`, `study_type`,
`intervention_type`, `lead_sponsor_class`, `lead_sponsor_name`, `country`,
`condition`, plus numeric `enrollment_count` / `start_year_numeric`), so adding a new
one is a two-line change — the compiler, aggregator, and selector all handle
arbitrary categorical dimensions generically.

## Running it

```bash
python -m venv venv
venv\Scripts\activate          # Windows  (source venv/bin/activate on macOS/Linux)
pip install -r requirements.txt
cp .env.example .env           # add ANTHROPIC_API_KEY (optional — see below)
uvicorn app.main:app --port 8000
```

Then open **http://127.0.0.1:8000/** for the UI, or POST to `/visualize` directly.

## How I validated correctness

This was the part I cared most about, because "it returns JSON" is not the same as
"the numbers are right."

- **Offline test suite (68 tests, `pytest`).** These run with **no network** against
  JSON fixtures captured from the real API (`tests/fixtures/`, generated by
  `scripts/capture_fixtures.py`). They cover schema round-tripping, enum normalization,
  defensive extraction, aggregation counts, the comparison/histogram/scatter/network
  builders, the full type-selection mapping, and citation attachment. A separate
  live-API smoke test is marked so it can be deselected offline (`-m "not live"`).

- **Deliberately messy fixtures.** I ran a `sparse_studies.json` with the
  defects real data actually has — missing `EnrollmentCount`, an empty interventions
  array, a `"January 2024"` date with no day — and asserted the extractors return
  `None` and never raise. Real CT.gov data is full of these; I wanted the failure mode
  to be "grouped as (unknown)", not a 500.

- **Cross-checking numbers against the source of truth.** For a network edge
  (Bristol-Myers Squibb → Ipilimumab in melanoma), I took the edge weight from the
  API response and independently recomputed it two ways: a scoped `/studies` count and
  a from-scratch pass over the raw records. All three agreed at **38**. Similarly, a
  phase-distribution bucket (`PHASE2` for melanoma = **1526**) matched the API's own
  server-side `AREA[Phase]PHASE2` count exactly. That's the check that convinced me the
  aggregation is faithful and not accidentally over/under-counting multi-valued fields.

- **Driving it live end-to-end** through the actual route for every chart type
  (distribution, time series, comparison, histogram, scatter, ranking, network,
  geographic) and confirming both paths — LLM and rule fallback — converge on the same
  counts.

## Reconciling the brief with the live API

A few things in the task's API description didn't match reality, and I only found out
by probing the live endpoint. I chose to follow the API and document the deviation
rather than ship something that 400s:

- **`filter.phase` isn't a real parameter.** The live API returns
  `400 "filter.phase is unknown parameter"`. Phase filtering actually goes through
  `filter.advanced` as `AREA[Phase](PHASE2 OR PHASE3)`. Verified by hand, then wired in.
- **`AREA[...]` names validate against metadata pieces, not search-area names.** The 19
  search-area names are `*Search` routing areas and don't include `StartDate`, yet
  `AREA[StartDate]RANGE[...]` works. So I validate AREA names against the metadata piece
  map (unioned with search areas).
- **The WAF blocks browser-like User-Agents.** Spoofing a Chrome UA gets a `403`;
  an honest `python-httpx`-flavored UA is admitted. This is the opposite of the usual
  advice, so it's documented right at the client's UA constant.

Each of these is commented at the relevant code site with the evidence, so the next
person doesn't have to re-discover it.

## Tools used, and what was hand-designed vs generated

The brief invites AI use and asks for honesty here, so:

- **AI pair programming (Claude / Claude Code)** was used heavily to scaffold the
  project, write boilerplate (Pydantic models, the httpx client, test skeletons), and
  draft the frontend. I treated it like a fast pair-programmer, not an oracle.
- **Live API probing** (curl / small async scripts) drove the parts that mattered —
  the schema of the introspection endpoints, the `filter.phase`/`AREA` discoveries, the
  correct multi-value separators, the WAF UA behavior. Those weren't guessed; they were
  measured.
- **The Anthropic SDK** for the interpreter (forced tool-use so the model must return
  a validated plan and nothing else).

**Designed deliberately (the load-bearing decisions were mine):**

- The core invariant (LLM writes only the plan) and the six-stage pipeline shape.
- The dimension-registry pattern that makes new group-by axes trivial and keeps
  extraction defensive in one place.
- The deterministic selector owning chart-type choice, so visualization type is a
  pure function of the plan and is unit-tested rather than model-dependent.
- The validation strategy — offline fixtures for logic, live cross-checks for the
  numbers, deliberately-broken fixtures for the messy-data paths.
- The deep-citations design (exact field-value excerpts, capped, with a true count).

**Generated then reviewed and adapted:** most of the mechanical code — schema classes,
the paginating client, chart-shaping helpers, and the test bodies — was AI-drafted and
then read line-by-line, corrected, and in several cases reworked once the live API
contradicted an assumption (the compiler and network builder both got rewritten after
probing revealed the real parameter behavior).

## Known limitations & honest trade-offs

- **Fetch cap.** Large result sets (e.g. 24k diabetes trials) are paginated up to a
  page cap, so counts for very broad queries reflect the fetched set and `truncated:
true` is set in `meta`. I chose an honest flag over silently partial numbers.
- **Multi-valued fields double-count by design.** A trial tagged both "Breast Cancer"
  and "Breast Neoplasms" contributes to both buckets — faithful to the raw arrays, but
  worth knowing.
- **Ranking by "sponsor" uses the lead sponsor name**, not collaborators, and the tail
  beyond the fetch cap isn't fully counted.
- **Rate limits.** Firing many multi-page queries in a burst can trip CT.gov's `429`;
  the client retries with backoff, but the UI can feel slow on big fetches.
- The `(unknown)` bucket carries **no citations** — there's no exact field value to
  quote when the field is missing, and I'd rather show nothing than fabricate one.

## What I'd improve with more time

- **True full-population counts.** Replace the page-cap with either the API's own
  server-side counting (an `AREA[...]` count per bucket, as I already do for spot
  cross-checks) or a streaming aggregate, so broad queries aren't `truncated`. This is
  the biggest correctness gap for very large result sets.
- **Condition/term normalization.** Merge synonymous conditions ("Breast Cancer" vs
  "Breast Neoplasms") via the API's MeSH/browse terms so ranking buckets aren't split.
- **Smarter fetch scoping.** Use `max_results` / server-side filtering more
  aggressively (e.g. push ranking's top-N to the API where possible) instead of
  fetching then aggregating locally.
- **Interpreter evals.** A small labeled set of question → expected-plan pairs run in
  CI, so prompt or model changes are measured, not vibes-checked. Right now the rule
  fallback is tested exhaustively but the LLM path is validated by live spot-checks.
- **Richer scatter/correlation.** Real per-study points are wired, but I'd add a
  computed correlation coefficient in `meta` and support more numeric axes.
- **Auth/robustness for a real deployment.** Rate-limit handling, request timeouts
  surfaced to the client, and pinning the frontend's CDN deps locally instead of
  loading from unpkg/esm.sh at runtime.
